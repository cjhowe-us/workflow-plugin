# discovery-contract

The workflow plugin builds a thin registry at session start. It records the existence and shape of
every installed workflow, artifact template, and artifact provider — but never loads bodies.

## Registry location

`$XDG_STATE_HOME/workflow/registry.json` (default `~/.local/state/workflow/registry.json`).

Regenerated by:

- `hooks/sessionstart-discover.sh` on session open.
- `scripts/discover.sh` on demand (e.g. after installing a new plugin mid- session).

## Registry schema

```json
{
  "generated_at": "2026-04-18T10:15:00Z",
  "entries": [
    {
      "scheme": "workflow" | "artifact-template" | "artifact-provider",
      "scope": "override" | "workspace" | "user" | "plugin",
      "path": "/abs/path/to/SKILL.md",
      "name": "<frontmatter name>",
      "description": "<frontmatter description>",
      "contract_version": 1
    }
  ]
}
```

Intentionally minimal. Consumers (`default` orchestrator, `/workflow` dashboard, conformance tests)
read this first to decide what to load next. The full SKILL.md body is only read when a specific
workflow/template is about to run.

## Scope precedence

Higher precedence wins when names collide:

| Scope     | Precedence | Location                                              |
|-----------|-----------:|-------------------------------------------------------|
| override  | 1 (top)    | `$CWD/.workflow-override/<kind>/<name>/`              |
| workspace | 2          | `$REPO/.claude/<kind>/<name>/` (committed to repo)    |
| user      | 3          | `~/.claude/<kind>/<name>/`                            |
| plugin    | 4 (bot.)   | `<installed-plugin>/skills/<kind>/<name>/`            |

`<kind>` in these paths is one of `workflows`, `artifact-templates`, `artifact-providers`.

Symlinks are followed. A workspace-scope workflow pointing at a checkout under `$HOME/work/...` is a
legal setup.

## Frontmatter-only rule

Discovery parses YAML frontmatter only. It never reads the body of any SKILL.md. Rationale:

- Fast session start — dozens of entries across many plugins parse in ms.
- Minimal context cost — only names + descriptions enter memory.
- Orchestrator decides on demand which specific workflow/template to load.

If a frontmatter field is missing or malformed, the entry is dropped with a warning; discovery is
resilient to one plugin's bad metadata.

## Lazy-load flow

1. `/workflow` invoked → orchestrator reads `registry.json` (small, fast).
2. Orchestrator interprets user intent against names + descriptions.
3. Once a specific workflow is chosen, orchestrator `Read`s its SKILL.md.
4. During dispatch, referenced sub-workflows + templates are loaded in turn.
5. Artifact providers are loaded only when a step needs to write / read an artifact they manage.

This keeps context cost proportional to the task, not to the installed breadth. Hundreds of
workflows can coexist without taxing a short task.

## Adding a new scope

The four scopes above are baked into the discovery hook. Adding another (e.g. `organization`,
`team`) requires editing `hooks/sessionstart-discover.sh` — which lives in the plugin and is
immutable to agents. Changes come via external PR.

## Debugging

```bash
# Show the current registry
jq . "$XDG_STATE_HOME/workflow/registry.json"

# Rebuild now
$(dirname "$(command -v claude)")/plugins/cache/workflow/*/scripts/discover.sh

# Check precedence resolution for a name
jq --arg n "<name>" '.entries | map(select(.name == $n))' \
  "$XDG_STATE_HOME/workflow/registry.json"
```
